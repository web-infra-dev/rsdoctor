我们很高兴地宣布 Rsdoctor 1.0 已经正式发布！经过一年的持续迭代与生产验证，我们正式发布 **Rsdoctor 1.0** —— 专为现代前端构建设计的深度分析工具。作为同时支持 Webpack 和 Rspack 的全链路观测平台，现已在多个大型项目中接入使用。

## 为什么 Rsdoctor

长期以来，构建分析一直面临以下难点和痛点：

1. **细节不够深入**：传统工具无法深入探索编译过程的细节和变化，开发者常常需要手动设置断点逐步调试，效率低下。
2. **数据割裂与孤立**：构建产物分析数据与构建扫描规则往往是分离的，缺乏统一的视图来综合分析和优化构建流程。
3. **现代框架支持不足**：现有方案对现代框架（如 Rspack）的支持不够全面，尤其是在分析 Rspack 内置 Loader 的能力上存在明显不足。

### 社区相关工具

在构建分析工具领域，有一些类似构建分析工具，例如：分析构建产物的工具 **webpack-bundle-analyzer**、**Statoscope** 工具，提供了不同的构建产物展示方式。构建编译工具方面有 **speedy-measure-plugin**。但以上工具还是无法更细粒度的展示构建细节及全方位的分析构建问题，以及无法对构建进行预警扫描。
基于以上原因，Rsdoctor 从中汲取灵感，结合多种工具的优点，致力于提供更细粒度、直观及全面的构建分析能力。

### Rsdoctor 诞生

根据以上社区工具现状，我们决定在社区经典工具的基础上开发一个专注于构建产物及编译分析的工具 —— **Rsdoctor**。

Rsdoctor 是以 Rspack 为核心实现的，我们为 Rsdoctor 设计了直观的用户界面和强大的分析能力，使其能够深入分析构建过程中的每一个细节。它不仅支持 Webpack 和 Rspack，还提供了对现代框架的全面支持。

在实现 Rsdoctor 的过程中，我们向社区中优秀的工具学习最佳实践，并聚焦于以下使用场景来设计 Rsdoctor：

- 作为一个全面的构建分析工具：帮助开发者深入了解构建过程中的每一个环节，提供详细的编译和产物分析。
- 作为一个可扩展的分析平台：允许开发者根据项目需求自定义分析规则，轻松实现对构建过程的衡量和优化。

通过 Rsdoctor，开发者可以更高效地识别和解决构建中的问题，从而提升项目的构建效率和质量。

## 能力支持

### 构建产物分析

- 我们在 **「Bundle Size」** 模块中，可以看到当前项目的产物数据信息概览，以及分析重复包引入的体积与上游依赖链路。

- 此外，还可以通过 **「Bundle Analysis」** 模块来进一步分析当前产物中资源与模块关系，体积数据等信息，以及查看模块引入的原因。

- 提供了重复包、ECMA version 检查等多项产物扫描规则。同时支持自定义规则扫描。

- 提供了 Bundle Diff 产物对比能力。

### 编译分析

面向 **Loaders、Plugins、Module Resolver** 提供了对应的数据与分析功能，来帮助你分析编译过程中的问题。

- 该模块主要提供了 **Rspack** 或 **Webpack Loaders** 内的输入输出、预估耗时、参数等数据分析的功能。

- 支持了拦截并收集 Plugins 的调用次数、预估耗时等数据信息。

- 同时，提供了项目内部单个文件中模块解析的路径数据以及预估耗时。Rspack 暂时未支持该模块。

### 可解决问题

以上能力中，我们可以用来解决下列常见的构建难点：

- 「构建好慢，想知道哪部分构建环节导致的？」
- 「构建后为什么没有按预期的进行代码替换或编译或产物结果为什么和预期不一致」
- 「我想优化 Initial chunk 但是不知道 Chunk 中包含了哪些模块，我该如何拆包」
- 「Monorepo 里，依赖问题，构建走的哪个版本的依赖」
- 「想知道 @ant-design/table/index.ts 为什么会被打包进来」
- 「想排查两次打包的变动，为什么引起线上性能的劣化或第三方包版本有什么变动」
- 「想查看构建重复包，并进行优化」

#### 🚀 Rspack 构建分析提效

通过在 Rspack 中集成 Rsdoctor 的原生插件，我们将 Rsdoctor 中耗时较大的数据处理逻辑前置到 Rspack 的构建阶段。这一改进显著提升了 Rspack 中 Rsdoctor 的构建分析效率，减少了整体分析时间。以下是配置示例：

```ts
import { RsdoctorRspackPlugin } from '@rsdoctor/rspack-plugin';

export default {
  plugin: [
    new RsdoctorRspackPlugin({
      experiments: {
        enableNativePlugin: true, // 开启 rsdoctor 的 rspack native plugin 特性加速构建分析
      },
    }),
  ],
};
```

#### 🔍 模块搜索功能

- **Search Module 能力**：在 `Bundle Size` 页面中，我们新增了模块搜索功能。用户可以通过输入模块名称来快速定位和查看模块所在的 `Assets`，从而更方便地分析模块的引用关系和大小。[详情](/guide/usage/bundle-size#%E6%A8%A1%E5%9D%97%E6%90%9C%E7%B4%A2%E5%8A%9F%E8%83%BD)

  示例图如下：

  <img src="https://assets.rspack.dev/others/assets/rsdoctor/search-modules.png" />

#### 🛠️ 新增构建扫描规则

- **Cross Chunks Package 规则**：新增了跨 Chunks 的重复包检测规则（E1002），该规则能够扫描不同 `chunks` 中的重复包。这些重复包可能导致打包代码冗余。

  解决方案可查看 [[E1002] Cross Chunks Packages](/guide/rules/rules#e1002-cross-chunks-package)

- **ECMA Version Check 规则**：增强了 ECMA 版本检测规则（E1004），用于检测不兼容的高级语法，[详情](/guide/rules/rules#e1004-ecma-version-check)。

  ```ts
  import { RsdoctorRspackPlugin } from '@rsdoctor/rspack-plugin';

  export default {
    plugin: [
      new RsdoctorRspackPlugin({
        linter: {
          rules: {
            'ecma-version-check': [
              'Warn',
              {
                ecmaVersion: 2015,
              },
            ],
          },
        },
      }),
    ],
  };
  ```

#### ⚙️ 新增配置项

- **output.compressData 配置**：我们新增了 `output.compressData` 配置项，用于控制是否对分析数据进行压缩处理。默认情况下，该选项为 `true`，即会对分析数据进行压缩。

  ```js
  new RsdoctorRspackPlugin({
    output: {
      compressData: true, // 是否压缩分析数据
    },
  });
  ```

#### Breakchange

插件配置中的 **reportCodeType** 和 **reportDir** 配置项移动到 output 下。[详情](/config/options/options#reportcodetype)。
